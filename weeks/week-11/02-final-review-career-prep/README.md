## CV/Resume Bullet Point Guide

### The Problem: Responsibilities vs Achievements

| ❌ Responsibility-Focused | ✅ Achievement-Focused |
|---------------------------|------------------------|
| "Built websites using HTML, CSS, and JavaScript" | "Developed 5 responsive landing pages using HTML5, CSS Grid, and JavaScript, improving mobile load speed by 40%" |
| "Worked on a team project" | "Collaborated with 3 developers on a React task manager app, delivering 2 sprints ahead of schedule using Git feature branching" |
| "Fixed bugs" | "Resolved 25+ cross-browser compatibility issues across Chrome, Firefox, and Safari, reducing support tickets by 60%" |
| "Used Git" | "Managed codebase versioning for a 5-person team using feature branches, pull requests, and code reviews on GitHub" |
| "Made websites accessible" | "Audited and remediated 30 pages to WCAG 2.2 AA standards, achieving a 100% Lighthouse accessibility score" |

### The CAR Method (Challenge → Action → Result)

Structure every bullet point around three components:

- **Challenge:** What problem did you face? (optional — use when context matters)
- **Action:** What did you do? Use strong action verbs.
- **Result:** What measurable outcome occurred? Include numbers if possible.

**Formula:** `[Action verb] [what you did] using [tools/technologies] resulting in [measurable impact].`

**Example:** "Redesigned the checkout flow using vanilla JavaScript and CSS animations, reducing cart abandonment by 22% and increasing conversion by $12K/month."

### Strong Action Verbs for Web Developers

| Category | Verbs |
|----------|-------|
| Building | Developed, Built, Architected, Engineered, Implemented, Created, Programmed |
| Improving | Optimised, Refactored, Enhanced, Migrated, Streamlined, Automated, Modernised |
| Fixing | Debugged, Resolved, Patched, Troubleshot, Fixed, Addressed |
| Collaborating | Collaborated, Coordinated, Partnered, Contributed, Led, Mentored |
| Analysing | Audited, Analysed, Evaluated, Benchmarked, Tested, Monitored |
| Designing | Designed, Prototyped, Wireframed, Mocked up, Styled, Responsified |

### Tailoring for Job Descriptions

1. Identify 3-5 keywords in the job description (e.g., React, TypeScript, Agile, REST APIs, CI/CD)
2. Ensure those exact keywords appear in your CV — but only if you genuinely have experience
3. Mirror the company's tone: startups want "built, shipped, hacked"; enterprise wants "implemented, coordinated, delivered"
4. Use an AI prompt: *"Given this job description: [paste], rewrite my CV bullet: [paste] to align with the role while remaining truthful."*

### Common Mistakes

- **Too generic:** "Responsible for building websites." → Be specific about stack, team size, impact.
- **No numbers:** "Improved performance." → Better: "Improved Lighthouse performance score from 45 to 92."
- **Passive voice:** "Was tasked with designing..." → Better: "Designed and prototyped..."
- **Too long:** Keep bullets to 1-2 lines. If it wraps three times, split or cut.
- **Lying or exaggerating:** Inflated claims are easily caught in interviews. Be honest.

## LinkedIn Profile Optimisation

### Headline (120 characters)
Don't just list your job title. Combine role + key skill + value proposition.

| ❌ Weak | ✅ Strong |
|---------|-----------|
| "Web Developer" | "Front-End Web Developer | React, TypeScript, Responsive Design | Building Accessible User Interfaces" |
| "Student" | "Aspiring Web Developer | HTML, CSS, JavaScript, React | Passionate About Accessible Web Experiences" |

### About Section (2,000 characters max; aim for 400-800)
Structure:
1. **Hook** (1-2 sentences): Who you are and what you do.
2. **Skills & expertise** (3-5 sentences): Key technologies, methodologies, achievements.
3. **Notable project / accomplishment** (1-2 sentences): Call out something impressive.
4. **Goal / call to action** (1 sentence): What you're looking for.

> **Example About Section:**
>
> "I'm a front-end web developer who builds fast, accessible, and human-friendly interfaces. I specialise in React, TypeScript, and modern CSS — and I believe great code is code that anyone can use.
>
> Over the past 6 months I've completed a 220+ hour AI Web Development program where I built 10+ projects including a AI-powered task manager and a fully responsive e-commerce landing page. I'm especially proud of achieving 98+ Lighthouse scores across every project through rigorous accessibility and performance audits.
>
> I'm currently seeking a junior front-end developer role where I can contribute to meaningful products while growing alongside experienced engineers."

### Featured Section
- Pin your top 3 projects. Each post should include:
  - A screenshot or GIF of the project
  - The tech stack (hashtags: #HTML #CSS #JavaScript #React)
  - A link to the live demo and GitHub repo
  - 2-3 sentences describing what it does and what you learned

### Skills & Endorsements
- List 15-20 relevant skills. Move your strongest to the top.
- Skills every web dev should have: HTML5, CSS3, JavaScript, React (or relevant framework), Responsive Design, Git, Accessibility (WCAG), REST APIs, Testing/Debugging.
- Ask classmates and instructors to endorse your top skills.

### Additional Profile Tips
- **Custom URL:** linkedin.com/in/yourname (edit in Settings → Edit public profile & URL)
- **Profile photo:** Professional headshot, plain or subtle background, good lighting
- **Background banner:** Custom banner with your tagline or portfolio URL (use Canva — 1584×396 px)
- **Open to Work:** Toggle on, set preferences (job titles, location, employment type)
- **Recommendations:** Ask 2-3 people (classmates, instructor, previous manager) to write brief recommendations
- **Activity:** Like, comment, and share relevant posts in the #webdev community — engagement boosts visibility

## Mock Interview Questions

### HTML/CSS

**Q1:** "Explain the difference between `display: none` and `visibility: hidden`."

- **Model answer:** `display: none` removes the element from the document flow — it takes no space and is not rendered. `visibility: hidden` hides the element visually but it still occupies space in the layout. Accessibility: `visibility: hidden` also hides from screen readers, while `display: none` removes it from the accessibility tree entirely.

**Q2:** "How would you create a responsive three-column layout that collapses to a single column on mobile?"

- **Model answer:** I'd use CSS Grid or Flexbox. With CSS Grid: `grid-template-columns: 1fr 1fr 1fr` on desktop, and a media query at, say, 768px that changes it to `grid-template-columns: 1fr`. I'd also use the `gap` property for spacing and consider using `minmax()` or `auto-fill` for more fluid responsive behaviour.

### JavaScript

**Q3:** "What is the event loop? Explain in simple terms."

- **Model answer:** JavaScript is single-threaded, meaning it can only do one thing at a time. The event loop is the mechanism that allows asynchronous code (like setTimeout, fetch, or DOM events) to work. It continuously checks if the call stack is empty, and if so, it takes the first task from the callback queue and pushes it onto the stack for execution. This is why code like `setTimeout(callback, 0)` runs after synchronous code.

**Q4:** "What's the difference between `==` and `===`?"

- **Model answer:** `==` (loose equality) performs type coercion before comparing, so `5 == '5'` is true. `===` (strict equality) checks both value and type without coercion, so `5 === '5'` is false. Best practice: always use `===` unless you have a specific reason to allow coercion.

**Q5:** "Explain how `this` works in JavaScript."

- **Model answer:** The value of `this` depends on how a function is called:
  - In a regular function: `this` refers to the global object (or `undefined` in strict mode).
  - In a method: `this` refers to the object the method was called on.
  - In an arrow function: `this` is lexically inherited from the enclosing scope (it doesn't have its own `this`).
  - With `call`, `apply`, or `bind`: `this` is explicitly set.

### Accessibility

**Q6:** "What are some ways you ensure a website is accessible?"

- **Model answer:** I follow WCAG 2.2 guidelines. Key practices: using semantic HTML (landmarks, headings hierarchy, proper form labels), ensuring colour contrast meets 4.5:1 for body text, adding alt text to images, supporting keyboard navigation with visible focus indicators, using ARIA roles only when native semantics aren't sufficient, and testing with tools like Lighthouse, axe DevTools, and a screen reader.

**Q7:** "What is the difference between `aria-label` and `aria-labelledby`?"

- **Model answer:** `aria-label` directly provides an accessible name as a string attribute value (e.g., `aria-label="Close dialog"` on a button with no visible text). `aria-labelledby` references the ID of another element on the page to use as the accessible name (e.g., `aria-labelledby="heading-id"`). Use `aria-labelledby` when you want to reuse visible text as the label; prefer `aria-label` for simple cases.

### Version Control

**Q8:** "You accidentally committed to `main` instead of a feature branch. How do you fix this?"

- **Model answer:** I'd use `git reset HEAD~1` to undo the commit on main while keeping the changes staged. Then `git checkout -b feature-branch` to create the feature branch, and commit there. If the commit was already pushed, I'd use `git revert` instead of reset to avoid rewriting remote history.

### Behavioural

**Q9 (STAR):** "Tell me about a time you faced a difficult technical challenge."

- **Situation:** "During my AI Web Development program, I was building a task manager app that used the OpenAI API for smart task suggestions."
- **Task:** "The API responses were sometimes slow or failed, which made the UI feel unresponsive and frustrated users."
- **Action:** "I implemented a loading skeleton, added error boundaries with React, and used `AbortController` to cancel stale requests. I also added retry logic with exponential backoff for transient errors."
- **Result:** "The app felt smooth even with slow API responses, and error rates dropped from 15% to under 1%. I learned a lot about handling async edge cases."

**Q10 (STAR):** "Describe a time you worked successfully as part of a team."

- **Situation:** "In a group project, 4 of us built a collaborative recipe-sharing platform in 2 weeks."
- **Task:** "We needed to coordinate our work without stepping on each other's code."
- **Action:** "I set up the GitHub repository with branch protection rules, defined a feature-branch workflow, and led code reviews to catch bugs early. We used daily standups to share progress."
- **Result:** "We completed the project on time with zero merge conflicts. Our instructor praised our Git discipline, and the app scored 98 on Lighthouse."
